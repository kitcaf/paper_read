# AI Paper Reader Web 应用需求文档

## 1. 文档目的

本文档用于重新定义项目第一阶段的产品形态与需求边界。当前结论是：

- 项目主入口应从 CLI 调整为 Web 应用
- CLI 不再作为核心用户使用方式，只保留给开发调试或运维脚本
- 产品设计要围绕“任务创建、过程可视化、结果复核、断点恢复”展开

本文档只覆盖 MVP 到第一版可用系统的核心需求，不展开到具体代码实现。

## 2. 产品定位

### 2.1 项目目标

构建一个面向研究者的 Web 应用，帮助用户完成两类高耗时工作：

1. 批量导入论文候选集，并基于标题与摘要进行 AI 粗筛
2. 对筛选后的论文执行 PDF 获取、全文解析、结构化抽取与结果沉淀

### 2.2 产品形态调整

第一版系统以浏览器使用体验为中心，用户应能通过页面完成以下动作：

- 创建研究项目
- 上传论文数据源
- 配置筛选标准与精读目标
- 发起后台任务
- 实时查看任务进度与失败原因
- 进入论文详情页查看筛选与精读结果

### 2.3 项目原则

- 以项目和任务为核心组织数据，而不是以命令行为核心
- 以“每篇论文可追踪、每个阶段可恢复”为核心能力
- 长耗时流程必须后台异步执行，不能阻塞页面操作
- Agent 负责编排与智能处理，Web 负责交互、状态展示和管理

### 2.4 非目标

以下内容不纳入第一阶段：

- 多租户权限系统
- 实时协作编辑
- 引文网络分析与知识图谱
- 自动生成完整综述文章
- 复杂的可视化工作流编排器

## 3. 核心用户场景

### 3.1 场景一：创建研究项目

用户进入 Web 应用后创建一个研究项目，填写研究主题、筛选标准、目标字段和输出要求。系统为该项目生成独立的数据空间与任务记录。

### 3.2 场景二：批量导入论文并执行粗筛

用户上传 `.csv` 或 `.json` 论文列表，系统完成字段校验后写入数据库。随后用户在页面点击“开始筛选”，系统在后台批量调用 LLM，生成 Keep 或 Discard 结论，并在页面持续更新进度、统计和失败信息。

### 3.3 场景三：查看筛选结果并启动精读

用户在结果页筛选出 Keep 论文，确认后发起“精读任务”。系统依次执行 PDF 下载、全文解析、结构化抽取，并将结果写入论文详情页与导出模块。

### 3.4 场景四：任务中断后恢复

当后台任务因网络、限流、解析失败或服务重启中断后，系统应保留已完成结果。用户重新进入项目页时，可以继续处理未完成或失败待重试的论文。

### 3.5 场景五：逐篇复核与导出

用户可以在 Web 页面逐篇查看：

- 论文元数据
- 摘要筛选理由
- PDF 与 Markdown 资产状态
- 精读抽取结果
- 单篇笔记与汇总导出

## 4. MVP 功能需求

## 4.1 项目管理模块

### 核心能力

- 创建研究项目
- 保存研究目标、筛选规则、精读字段配置
- 展示项目概览、任务统计和最近活动

### 输出结果

- `project`
- `task_run`
- 项目级配置记录

## 4.2 论文导入模块

### 输入要求

支持上传本地 `.csv` 或 `.json` 文件，至少包含：

- `paper_id`
- `title`
- `abstract`
- `source_url` 或 `doi`

### 核心能力

- 页面上传与解析文件
- 校验必填字段
- 对重复论文做去重或覆盖策略
- 将导入结果反馈到页面

### 输出结果

- 标准化论文记录
- 导入日志
- 错误行列表

## 4.3 摘要筛选模块

### 核心能力

- 用户在项目页发起筛选任务
- 后端将任务提交给内置 agent 模块处理
- 批量读取标题与摘要执行判断
- 页面展示实时进度、成功数、失败数和 Keep 数
- 支持按状态、分数、结论进行筛选与排序

### 结构化输出要求

每篇论文至少包含：

- `relevance_score`
- `decision`
- `reasoning`
- `status`
- `model_name`
- `processed_at`

## 4.4 精读与抽取模块

### 核心能力

- 仅对 Keep 论文发起精读任务
- 下载 PDF 并保存文件资产
- 解析全文为 Markdown
- 根据项目配置抽取结构化结果
- 生成单篇笔记与可读摘要

### 结构化输出示例

- `research_problem`
- `core_method`
- `datasets`
- `baselines`
- `main_results`
- `limitations`
- `evidence_spans`

## 4.5 任务中心模块

### 核心能力

- 查看任务队列、当前阶段、执行时间和失败原因
- 展示任务级与论文级进度
- 支持失败重试
- 支持中断后恢复

### 页面要求

- 项目概览页展示聚合统计
- 任务详情页展示阶段事件流
- 论文详情页展示单篇状态与结果

## 4.6 导出模块

### 核心能力

- 导出筛选结果为 CSV
- 导出精读结果为结构化表格
- 导出单篇 Markdown 笔记
- 导出项目级汇总结果

## 5. 页面与交互要求

### 5.1 必需页面

- 登录页或入口页（如第一版不做鉴权，可保留空壳）
- 项目列表页
- 项目详情页
- 论文列表页
- 论文详情页
- 任务中心页
- 导出页

### 5.2 关键交互要求

- 页面不得依赖刷新才能看到最新状态
- 长任务状态应通过轮询、SSE 或 WebSocket 实时更新
- 对失败论文要能直接展示错误原因
- 对 Keep 论文要支持快速进入详情页复核

## 6. 非功能性要求

### 6.1 可恢复性

- 单篇论文失败不影响整批任务
- 已完成结果不可因任务重启而丢失
- Agent 任务必须支持 checkpoint 与恢复

### 6.2 可追踪性

- 每篇论文的每个阶段都要有状态记录
- 每次任务执行都要保留日志与事件
- 关键产物路径必须可追溯

### 6.3 可维护性

- 前端、后端、Agent 模块边界清晰
- LLM、PDF 下载器、PDF 解析器都应可替换
- Prompt 规则、结构化字段定义、导出模板应解耦

### 6.4 性能与体验

- Web 首屏只加载项目与任务摘要，不直接加载大文件内容
- 长耗时任务后台执行
- 任务列表、论文列表支持分页和筛选

## 7. 推荐的一期系统边界

第一版建议只做单组织或单用户场景，但架构上使用 Web 方式组织，避免后续从 CLI 重构为服务化系统时再次推翻。

推荐的一期能力边界如下：

- 浏览器端 Web 应用
- TypeScript 后端 API 服务
- 后端内置 TypeScript Agent 模块
- PostgreSQL 持久化
- 本地文件存储或对象存储

## 8. MVP 开发优先级

### 第一优先级

- 项目管理
- 论文导入
- 摘要筛选任务
- 任务进度页
- PostgreSQL 持久化
- 结果导出

### 第二优先级

- PDF 下载
- Markdown 解析
- 全文结构化抽取
- 论文详情页

### 第三优先级

- 人工复核节点
- 更细粒度的任务事件流
- 权限体系
- 对象存储与部署增强

## 9. 结论

这个项目更适合被设计成“Web 管理台 + 后端内置 agent 模块系统”，而不是继续围绕 CLI 命令推进。

原因很明确：

- 任务是长耗时的，Web 更适合展示进度、失败原因和结果沉淀
- 用户操作天然是项目式、任务式和列表式，不是命令式
- 后续加入复核、重试、导出、权限时，Web 架构扩展成本更低

因此，后续设计和开发都应以 Web 应用为主线展开。
